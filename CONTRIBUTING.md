# Contributing to Gale

We welcome contributions! This document outlines the process for contributing to Gale.

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/your-username/Gale.git`
3. Add the upstream remote: `git remote add upstream https://github.com/GaleMC/Gale.git`
4. Run the initial build: `./gradlew applyAllPatches`

## Understanding the Project Structure

Gale is a fork of [Paper](https://github.com/PaperMC/Paper). The project uses Paperweight, Paper's patching system.

Key directories:
- `gale-server/src/minecraft/java/` - Minecraft source code (generated by patches, do not edit directly)
- `gale-server/minecraft-patches/sources/` - Individual source file patches
- `gale-server/minecraft-patches/features/` - Feature patches (larger changes spanning multiple files)
- `gale-api/` - The Gale API module
- `build-data/` - Build configuration data

## How to Add or Modify Patches

### Source File Patches (recommended for small changes)

1. Make your changes in `gale-server/src/minecraft/java/`
2. Run `./gradlew rebuildMinecraftSourcePatches` to regenerate patches
3. Review the generated patch files in `gale-server/minecraft-patches/sources/`
4. Test your changes: `./gradlew applyMinecraftSourcePatches`

### Feature Patches (for larger changes)

1. Create a new file in `gale-server/minecraft-patches/features/`
2. The file should contain the full implementation of your feature as a diff against the Paper base
3. Test your changes: `./gradlew applyAllPatches`

## How to Add Access Transformers

Access transformers allow you to access private or protected members of Minecraft classes.

1. Edit `build-data/gale.at`
2. Add your access transformer entry following the existing format
3. Test your changes: `./gradlew applyAllPatches`

For more information on access transformers, see Paper's [Access Transformers documentation](https://docs.papermc.io/paper/dev/access-transformer-system).

## Commit Message Guidelines

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests after the first line

## Pull Request Process

1. Create a new branch for your changes
2. Make your changes and commit them
3. Push your branch to your fork
4. Open a pull request against the `main` branch
5. Ensure all CI checks pass
6. Wait for review

## Code Style

- Try to minimize changes to Minecraft and Paper classes, if possible
- Follow the existing code style in the project
- Use 4 spaces for indentation (no tabs)
- Use American English spellings (optimize, organize, color)

## AI and Tooling

- Do not submit code you do not fully understand
- Do not submit obfuscated code

## Testing

- Ensure your changes compile: `./gradlew applyAllPatches`
- Verify complex code with tests
- Measure performance improvement in practice, if it is not clear from code alone
- Run tests if applicable: `./gradlew test`
- Verify your patches apply cleanly: `./gradlew applyMinecraftSourcePatches --stacktrace`
